# FAQ

## Purpose

Frequently asked questions about Hackathon Foundation.

---

### What is Hackathon Foundation?

Hackathon Foundation is a free, open-source, AI-agnostic engineering framework that helps students and developers build hackathon projects faster by organizing AI coding assistants through structured roles, shared context, rules, skills, templates, and workflows.

### How is this different from other AI coding frameworks?

Most frameworks focus on prompts or model configuration. Hackathon Foundation focuses on **structure** — it treats the repository as a company, the user as the CEO, and the AI as employees with defined roles. It is not tied to any specific AI model or coding assistant.

### Is it really free?

Yes. See [FREE_FOREVER.md](./FREE_FOREVER.md). The project references only free AI models, free tools, and free APIs. There is no paid tier, no premium features, no subscription.

### Do I need to pay for AI models to use this?

No. All recommended models in [FREE_MODELS.md](./FREE_MODELS.md) have free tiers or are open-source and runnable locally.

### Which AI coding assistants work with Hackathon Foundation?

The framework is AI-agnostic. It works with any AI coding assistant — OpenCode, Gemini CLI, Continue, Cline, Roo Code, VS Code Copilot, and others. See [INTEGRATIONS.md](./INTEGRATIONS.md) for details.

### Do I need to be an experienced developer to use this?

No. The framework is designed to be beginner-friendly. The structure guides you through what to do at each step. The documentation explains every concept.

### What if I am working alone?

The framework works for solo developers too. You are the CEO of a company of one. You assign tasks to AI roles — one session you are the Software Architect, the next you are the Frontend Engineer.

### What if I am working in a team?

The framework supports teams. Different team members can handle different CEO responsibilities — one manages architecture, another manages frontend tasks, another manages testing.

### Does this repository contain code?

No. It contains structure — documentation, definitions, templates, and guides. It is language-agnostic and stack-agnostic. You use it alongside your actual project code.

### Can I use this with my existing project?

Yes. The framework is modular. You can adopt only what you need — roles, rules, skills, templates — without restructuring your project.

### How do I start?

1. Clone the repository.
2. Read [VISION.md](./VISION.md) and [MISSION.md](./MISSION.md).
3. Read the [docs/README.md](./README.md) to understand the documentation structure.
4. Choose a role from [ENGINEERING_ROLES.md](./ENGINEERING_ROLES.md).
5. Follow the workflow in [WORKFLOW_OVERVIEW.md](./WORKFLOW_OVERVIEW.md).
6. Select a model from [MODEL_SELECTION_GUIDE.md](./MODEL_SELECTION_GUIDE.md).
7. Start building.

### How do I contribute?

See [CONTRIBUTING.md](./CONTRIBUTING.md). Contributions of all sizes are welcome — from typo fixes to new role definitions.

### What license is the repository under?

MIT License. See the LICENSE file in the repository root.

### Can I use this for commercial projects?

Yes. The MIT License permits commercial use. The framework does not require any attribution or payment.

### How is the project governed?

See [COMMUNITY.md](./COMMUNITY.md). The project is community-driven with maintainers managing the repository. Major decisions are made through consensus in public discussions.

### What if a model I use becomes paid?

The framework is model-agnostic. If a model becomes paid, you switch to another free model. The structure, roles, rules, and skills remain the same regardless of which model executes them.

### What if a tool I use is not listed in INTEGRATIONS.md?

The integration philosophy extends to any tool. Read [INTEGRATIONS.md](./INTEGRATIONS.md) for the general approach, then apply it to your tool. Contributions for new integration guides are welcome.

### Is there a community chat or forum?

All community communication happens through GitHub Issues and Discussions. See [COMMUNITY.md](./COMMUNITY.md).

### I found an error in the documentation. What should I do?

Open a GitHub issue describing the error, or submit a pull request with the fix. See [CONTRIBUTING.md](./CONTRIBUTING.md).

### I have an idea for a new role/skill/template. What should I do?

Open a GitHub issue to discuss the idea. If the community agrees, follow the contribution workflow to implement it.

For more detailed answers, browse the table of contents in [docs/README.md](./README.md) or search for the relevant document.
