# Community

## Purpose

This document defines the Hackathon Foundation community — how it operates, who participates, and how members interact.

## Community philosophy

Hackathon Foundation is built on the belief that **great software engineering frameworks should be free, open, and community-driven.** The community is not an audience — it is a network of contributors who make the framework better for everyone.

The community philosophy:

> Everyone is a potential contributor. Every contribution matters. Every contributor is respected.

## Who the community serves

The community serves three groups:

| Group | Need |
|---|---|
| **Students** | A free, structured framework for learning AI-assisted development and building hackathon projects. |
| **Hackathon participants** | A reliable template that reduces chaos and increases project quality under time pressure. |
| **Developers** | A professional-grade engineering structure for AI-assisted projects, regardless of budget. |

## How the community operates

### Open participation

Anyone can participate. There are no membership requirements, no approval processes, and no barriers to entry.

### Merit-based influence

Influence in the project is earned through contribution quality, not status. A well-written skill definition from a first-time contributor carries the same weight as one from a maintainer.

### Transparent decision-making

Decisions about the repository are made in public. Issues, pull requests, and discussions are visible to everyone. There are no private decision channels.

### Consensus-driven

Major changes — new roles, structural changes, principle changes — are discussed until consensus emerges. Minor changes — typo fixes, small refinements — are made directly.

## Community roles

| Role | Responsibilities |
|---|---|
| **User** | Uses the framework for hackathons and projects. Provides feedback through issues. |
| **Contributor** | Submits pull requests for new content, refinements, or fixes. |
| **Reviewer** | Reviews pull requests for quality, consistency, and standards compliance. |
| **Maintainer** | Manages the repository, reviews contributions, makes final decisions, and ensures project health. |
| **Steering Committee** | (Future) Guides the long-term direction of the project. |

Everyone starts as a user. Contribution leads to contributor status. Consistent quality contributions may lead to reviewer or maintainer roles.

## Communication channels

| Channel | Purpose |
|---|---|
| GitHub Issues | Bug reports, feature requests, discussions about specific changes. |
| GitHub Discussions | General questions, ideas, community conversation. |
| Pull Requests | Specific contributions and their review. |
| README | Project overview and entry point. |

There is no official chat room, forum, or mailing list. All communication happens through GitHub. This ensures that every discussion is archived, searchable, and accessible.

## Community standards

### For users

- Search existing issues before creating new ones.
- Provide clear, specific feedback.
- Be respectful in all interactions.

### For contributors

- Follow the contribution workflow in [CONTRIBUTING.md](./CONTRIBUTING.md).
- Ensure contributions meet the repository standards.
- Respond to review feedback promptly.

### For reviewers

- Review contributions against the criteria in [CONTRIBUTING.md](./CONTRIBUTING.md).
- Provide constructive, specific feedback.
- Approve contributions that meet the standards.

### For maintainers

- Triage issues and pull requests promptly.
- Ensure the repository standards are upheld.
- Guide the project direction.
- Foster a welcoming community.

## Code of conduct

All community members must follow the [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md). Harassment, discrimination, and disrespectful behavior are not tolerated.

## Community growth

### Phase 1 — Foundation (current)

The repository is built by the initial maintainers. The community is small. Contributions are reviewed directly.

### Phase 2 — Contribution

Early adopters contribute roles, skills, templates, and refinements. Contribution guidelines are tested and refined. Reviewers are identified from consistent contributors.

### Phase 3 — Self-sustaining

The community has active reviewers and maintainers. Most contributions are reviewed and merged by community members. The initial maintainers step back to a steering role.

### Phase 4 — Ecosystem

The community produces third-party content — integrations, translations, specialized role packs. Hackathon Foundation is adopted as a standard template in hackathons and educational settings.

## How to get involved

1. **Use the framework.** Build a hackathon project with it. Report what worked and what did not.
2. **Open an issue.** Found a gap? Have an idea? Open an issue to discuss it.
3. **Submit a pull request.** Ready to contribute? Follow the workflow in [CONTRIBUTING.md](./CONTRIBUTING.md).
4. **Review contributions.** Help review open pull requests. Fresh perspectives catch what maintainers miss.

For the contribution workflow and standards, see [CONTRIBUTING.md](./CONTRIBUTING.md). For the project's long-term direction, see [ROADMAP.md](./ROADMAP.md).
