# Communication Policy

## Purpose

Define the rules for all communication between roles within the Hackathon Foundation framework. Communication must be clear, structured, and auditable.

## Rule 1: All communication must be written

All role-to-role communication must be written in a shared, persistent medium. Verbal or transient communication (such as unsaved chat messages) is not sufficient for decisions, specifications, or handoffs. Written communication includes:

- Files committed to the repository.
- Structured messages in a shared task system.
- Documents in the project's communication channel.

## Rule 2: Communication must follow a standard format

Every communication between roles must include:

- **Sender** — The role sending the communication.
- **Recipient** — The role receiving the communication.
- **Purpose** — What this communication is about (request, report, handoff, clarification).
- **Body** — The content of the communication, structured for the recipient's needs.
- **Deadline** — When a response is needed (if applicable).
- **Priority** — Low, medium, high, or urgent.

## Rule 3: Progress reporting rules

Any role working on a task must provide progress reports at the following intervals:

- Every session end — Summary of what was accomplished.
- Every handoff — Full status of the task.
- On request — Immediate status when asked by the CEO.

Each progress report must include:

- **Status** — Not started, in progress, blocked, complete.
- **Completed** — What was finished since the last report.
- **Next** — What will be worked on next.
- **Blockers** — Anything preventing progress (or "none").
- **Confidence** — High, medium, low — how confident the role is in meeting the deadline.

## Rule 4: Blocker reporting rules

When a role encounters a blocker, the following rules apply:

- **Immediate report** — The blocker must be reported within the same session it is discovered.
- **Blocker format** — The report must include:
  - What is blocked.
  - The exact obstacle.
  - What has been tried to resolve it.
  - What is needed to unblock (specific decision, information, or resource).
  - Impact estimate — How long the delay will be if not unblocked.
- **No silent waiting** — A blocked role must not remain idle. If waiting for a response, the role must work on an alternative task or explicitly request direction from the CEO.

## Rule 5: Clarification requirements

When any instruction or specification is unclear, the receiving role must:

1. **Identify the specific ambiguity** — Quote the unclear portion.
2. **State the interpretation** — What the role believes the instruction means.
3. **Request confirmation** — Ask the sender to confirm or correct.
4. **Do not proceed past the ambiguity** — Work that depends on the unclear instruction must not start until clarification is received.

It is a violation to proceed with an unclear instruction based on assumption.

## Rule 6: No assumptions about shared context

A role must never assume that another role has context it has not explicitly received. When communicating, the sender must include or reference all context necessary to understand the message.

Common violations:

- Assuming the recipient has read related files.
- Assuming the recipient remembers prior decisions.
- Assuming the recipient understands domain-specific terminology without explanation.

## Rule 7: Response time expectations

| Priority | Response expected within |
|---|---|
| Urgent | Same session |
| High | Next session |
| Medium | Within 2 sessions |
| Low | Within 5 sessions |

If a response cannot be provided within the expected time, the recipient must acknowledge receipt and provide an estimated response time.

## Enforcement

Communication violations that cause delays or errors are reported to the CEO. Repeated violations may result in the role being required to have all communications reviewed before sending.
