# Documentation Standards

## Purpose

This document defines the standards for writing documentation within the Hackathon Foundation framework. Consistent documentation ensures that all content is readable, findable, and maintainable.

## Markdown style

### Headings

- Use ATX-style headings (`#`, `##`, `###`).
- Include a space after the `#` characters.
- Use sentence case for headings (only the first word and proper nouns are capitalized).
- Do not skip heading levels (e.g., do not go from `##` to `####`).
- Every document begins with a single `#` title heading.

```
# Document Title

## Section Heading

### Subsection Heading
```

### Emphasis

- Use `**bold**` for emphasis on key terms.
- Use `*italic*` for introducing new terms or mild emphasis.
- Use `` `code` `` for file names, paths, commands, and code references.
- Use `---` for horizontal rules between sections.

### Lists

- Use `-` for unordered lists.
- Use `1.` for ordered lists.
- Capitalize the first word of each list item.
- Use consistent punctuation (either all items end with a period, or none do).

### Links

- Use relative paths for internal links: `[text](./target.md)`.
- Use full URLs for external links: `[text](https://example.com)`.
- Link text should describe the target: `See [mission.md](./mission.md)` not `See here`.

### Code blocks

- Use fenced code blocks with language annotation.
- Use ` ``` ` on their own line before and after the code block.
- Specify the language after the opening ` ``` `.

```
```python
def hello():
    print("Hello, world!")
```
```

## Document structure

Every document should follow this structure:

```
# Title

## Purpose

One to two sentences explaining why this document exists.

## Content

The main content of the document.

## Connection to other documents

Links to related documents at the end.
```

### Purpose section

Every document must begin with a Purpose section after the title. This section answers: "Why does this document exist?" and "Who should read it?"

### Content sections

Organize content into logical sections with descriptive headings. Use subsections for related groups of information.

### Connection section

End with links to related documents. This ensures readers can navigate to related content.

```
For the mission that drives this document, see [mission.md](./mission.md).
```

## Writing style

### Be clear

- Prefer short sentences.
- Use active voice: "The AI generates output" not "Output is generated by the AI."
- Avoid jargon unless defined in the glossary.
- Define acronyms on first use.

### Be concise

- Use the minimum number of words needed to convey the meaning.
- Remove filler phrases like "It is important to note that."
- Use bullet points and tables to organize information.

### Be specific

- Use concrete examples instead of abstract descriptions.
- Include file paths and names when referencing repository content.
- Use exact numbers and names instead of "some" or "several."

### Be consistent

- Use the same terminology throughout all documents.
- Use the same formatting for similar content types.
- Use the same document structure across all files.

## Tables

Use tables to present structured comparisons or lists.

```
| Column A | Column B | Column C |
|---|---|---|
| Value 1 | Description 1 | Note 1 |
| Value 2 | Description 2 | Note 2 |
```

- Use alignment consistently (default left-aligned is preferred).
- Keep tables simple. Complex tables should be split into multiple simpler tables.

## Cross-references

Every document must cross-reference related documents:

- At the end of each document, link to related documents.
- Within the text, link to relevant sections using relative paths.
- When referencing a concept defined in another document, link to that document.

## Examples

Where helpful, include examples:

- Code examples show how something is used.
- Before/after examples show the effect of a change.
- Decision examples show the reasoning process.

## File naming

- Use kebab-case for all file names: `documentation-standards.md`.
- Use lowercase only.
- Use `.md` extension for all markdown files.
- File names should describe the content.

## What to document

Document when:

- A decision is made (record the decision and rationale).
- A component is created (explain its purpose and usage).
- A process is defined (describe the steps).
- Something changes (update existing documentation).

Do not document when:

- The information is temporary and will change immediately.
- The audience is only the author (personal notes belong elsewhere).
- The information is obvious from the code itself.

## Connection to company culture

These documentation standards support the Documentation First value in [company-culture.md](./company-culture.md). Documentation is not an afterthought — it is a primary deliverable with its own quality standards.
