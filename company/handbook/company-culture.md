# Company Culture

## Purpose

Company culture defines the values that guide every decision. These values are not suggestions — they are the criteria against which every output, every role definition, and every policy is measured.

## Core values

### Quality over speed

Building something that works is not enough. It must be correct, maintainable, and well-documented. Speed is important in a hackathon, but quality determines whether the project survives beyond the event.

- Always prefer a correct, simple solution over a fast, fragile one.
- Review before accepting. No output is accepted without review.
- Documentation is part of quality, not an optional add-on.

### Think before coding

The most expensive bugs are the ones caused by not thinking before writing code. Architecture, design, and planning happen before implementation.

- Understand the problem before proposing a solution.
- Design before implementing.
- Review the architecture before writing the first line of code.

### Documentation first

Documentation is not an afterthought. It is a primary deliverable. Code without documentation is incomplete.

- Every component, API, and architecture decision must be documented.
- Documentation is written alongside code, not after.
- Documentation is reviewed with the same rigor as code.

### Reuse before creating

Before building something new, check if it already exists. The framework is designed to be modular and reusable. Creating duplicates wastes time and creates maintenance burden.

- Check existing assets before creating new ones.
- Extend existing work when possible instead of replacing it.
- If something does not exist, consider whether it should be added to the shared assets.

### Small, incremental improvements

Large changes introduce large risks. Small, incremental changes are easier to review, test, and debug.

- Prefer small, focused commits over large, sweeping changes.
- Each change should do one thing and do it well.
- Build features iteratively, not all at once.

### Respect engineering standards

Consistency is more important than personal preference. The framework defines standards for a reason — they ensure that all output is consistent regardless of which AI model or human produced it.

- Follow the defined rules in `company/policies/`.
- Use the defined templates in `company/assets/`.
- Respect the naming conventions and folder structure.

### Free Forever

Free is a feature, not a limitation. Every tool, model, and resource recommended must be free to use. This is non-negotiable.

- Never recommend a paid tool when a free alternative exists.
- If no free alternative exists, document the gap rather than recommending a paid solution.
- Challenge every assumption that a paid tool is necessary.

### Community driven

The framework is built by the community, for the community. Every contribution is valued. Every user is a potential contributor.

- Design for contribution. Make it easy for others to add and improve.
- Be welcoming to new contributors.
- Share knowledge freely.

## How values apply

When making any decision — whether writing code, defining a role, or creating a policy — ask:

1. Does this prioritize quality over speed?
2. Have I thought about this before implementing?
3. Is the documentation complete?
4. Have I checked if this already exists?
5. Can this be done as a smaller change?
6. Does this follow the defined standards?
7. Is everything free?
8. Would this be easy for someone else to understand and contribute to?

If the answer to any of these questions is no, the decision needs to be reconsidered.

## Connection to mission

These values exist to fulfill the mission defined in [mission.md](./mission.md). The mission defines *what* we do. The culture defines *how* we do it.
